Hashing Strategy for Sharding

Assign record with key k to node: node = h(k) % n h(k) = hash function (e.g., MD5, SHA) n = number of nodes Ensures even distribution of records across nodes. Problem: adding/removing nodes requires rehashing → Consistent Hashing solves this.

MapReduce

A programming model for processing large datasets in parallel across a cluster.

1
Input: A bag of (key, value) pairs.
2
Map: Each worker processes its chunk and emits intermediate (key, value) pairs.
3
Shuffle: All intermediate values with the same key are grouped together and sent to the same reducer.
4
Reduce: Aggregates the values for each key into a final result.
5
Output: Final (key, value) pairs written to storage.

Word Count Example (classic MapReduce)

Map: (doc_id, "hello world hello") → [("hello",1), ("world",1), ("hello",1)] Shuffle: ("hello", [1,1]), ("world", [1]) Reduce: ("hello", 2), ("world", 1)

HDFS (Hadoop Distributed File System)

  • Write-once, read-many model — files are immutable once written.
  • Fixed-size block partitioning — large files split into blocks (default 128 MB).
  • 3-way replication — each block replicated on 3 separate nodes for fault tolerance.
  • NameNode: master — stores metadata (file names, block locations).
  • DataNode: worker — stores actual data blocks.

Limitations of MapReduce / Why Spark?

MapReduceSpark
Writes intermediate results to disk (slow)Keeps data in memory (10–100× faster for iterative)
Only Map + Reduce primitivesRich transformation library (100+ ops)
No support for iterative algorithmsEfficient for ML, graph, streaming
Java-heavy APIPython, Scala, Java, R APIs

Dask

  • Python-native parallel computing — extends NumPy/Pandas to datasets larger than RAM.
  • Lazy evaluation: builds a task graph, executes only on .compute().
  • Block algorithms: splits computation into chunks (~100 MB – 1 GB recommended).
  • Rechunking is expensive — choose chunk size wisely upfront.

Apache Spark

Core Concepts

RDD (Resilient Distributed Dataset)

  • Fundamental data structure in Spark
  • Immutable — create new RDD instead of modifying
  • Distributed — partitioned across cluster nodes
  • Fault-tolerant via data lineage (re-compute lost partitions)
  • Lazy evaluation — transformations not executed until an action is called

DataFrame

  • Organized into named columns (like a table)
  • Supports nested types: StructType, ArrayType, MapType
  • Optimized by Catalyst optimizer
  • DSL similar to Pandas
  • Also lazy — needs an action to execute

Transformations vs Actions

CategoryDescriptionExamples
Transformation Lazy — returns a new RDD/DF; no computation yet map(), filter(), flatMap(), join(), union(), reduceByKey(), groupByKey(), distinct(), sortBy()
Action Eager — triggers computation, returns a value or writes data collect(), count(), show(), save(), first(), take(n)

Narrow vs Wide Transformations

Narrow

  • Each input partition → exactly one output partition
  • No network shuffle needed
  • Examples: filter, map, flatMap
  • Faster, can pipeline stages

Wide (Shuffle)

  • Multiple input partitions → output partitions
  • Requires network communication (shuffle)
  • Examples: groupByKey, sortBy, join
  • Expensive — stage boundary in DAG

Catalyst Optimizer (DataFrame query engine)

  • DataFrame operations build an Abstract Syntax Tree (AST) of the logical plan.
  • Catalyst rewrites and optimises the plan before execution (predicate pushdown, column pruning, join reordering).
  • This is why DataFrames are faster than raw RDDs for structured data.
employees.join(dept, employees.deptId == dept.id) .where(employees.age > 30) .groupBy("deptName") .agg(count("*")) # Catalyst pushes the .where() filter BEFORE the join — reads less data

Partitioning & Join Optimisation

  • partitionBy(key) — partition RDD so all records with the same key land on the same node.
  • If two RDDs are partitioned by the same key, join() becomes a narrow transformation (no shuffle).
  • Mismatched partitioning → all-to-all network communication (expensive wide transformation).
  • Use HashPartitioner for key-based partitioning; RangePartitioner for sorted ranges.

Inverted Index (Take-Home Exercise)

Classic IR structure: word → list of document IDs that contain it.

# Input format: "D1 || word1 word2 word3 ..."
lines = spark.textFile("documents.txt")

inv_index = lines \
    .flatMap(lambda line: [
        (word, line.split("||")[0].strip())
        for word in line.split("||")[1].strip().split()
    ]) \
    .distinct() \
    .groupByKey() \
    .mapValues(lambda docs: sorted(list(docs))) \
    .sortByKey()

inv_index.collect()
# [('data', ['D1','D2']), ('science', ['D1','D3']), ...]
Partitioning tip: Use repartition() when you need to increase partitions (shuffle). Use coalesce() to reduce partitions without shuffle. Always partition by join keys to avoid cross-node shuffles.

3

Porter's Stemmer — 5 Phases (sequential rules)

  • Phase 1: Remove plurals & past tenses (sses→ss, ies→i, s→∅)
  • Phase 2: Remove suffixes (ational→ate, tional→tion)
  • Phase 3: Further suffix removal (icate→ic, alize→al)
  • Phase 4: Remove derivational suffixes (ement, ment, ent, ion→if preceded by s/t)
  • Phase 5: Clean up (remove trailing e)
Stop words problem: Removing stop words blindly can destroy meaning. Examples: "President of University of Hong Kong" (removing "of" breaks the entity), "Let it be" (song title destroyed), "flights to London" (destination lost). Solution: use Named Entity Recognition (NER) to protect important phrases before stop-word removal.

Text Feature Representations

One-Hot Encoding

  • Each word → binary vector of vocabulary size
  • Only the position of that word is 1, rest 0
  • Sparse; no notion of word similarity
  • Example: [Fish, Dog, Cat] → [1,0,0], [0,1,0], [0,0,1]

Bag of Words (BoW)

  • Count occurrences of each word in document
  • Word order is lost
  • Sparse representation
  • Can be weighted by TF-IDF for importance

TF-IDF — Full Formula with Worked Example

TF: how many times term t appears in document d. IDF: penalises terms common across many documents.

IDF(t, D) = log( (|D|+1) / (DF(t,D)+1) ) ← +1 = Laplace smoothing TF-IDF(t, d, D) = TF(t, d) × IDF(t, D) |D| = total docs; DF(t,D) = docs containing t; High TF-IDF = frequent in ONE doc, rare elsewhere

Worked Example (|D|=2)

D0="Python python Spark Spark", D1="Python SQL" TF(python,D0)=2 TF(spark,D0)=2 TF(python,D1)=1 TF(sql,D1)=1 DF(python)=2 DF(spark)=1 DF(sql)=1 IDF(python) = log(3/3) = 0 ← in BOTH docs → not discriminating IDF(spark) = log(3/2) ≈ 0.405 IDF(sql) = log(3/2) ≈ 0.405 TF-IDF(python,D0) = 2×0 = 0 TF-IDF(spark, D0) = 2×0.405 = 0.811 ← "spark" is keyword for D0 TF-IDF(sql, D1) = 1×0.405 = 0.405 ← "sql" is keyword for D1

Levenshtein (Edit) Distance

Minimum number of single-character edits to transform one string to another.

Operations: Insert, Delete, Substitute (each costs 1)

Recurrence: If s[i] == t[j]: dp[i][j] = dp[i-1][j-1] (no cost) Else: dp[i][j] = 1 + min( dp[i-1][j], # delete from s dp[i][j-1], # insert into s dp[i-1][j-1] # substitute )

Example: "kitten" → "sitting" = 3 edits (k→s, e→i, insert g)

Byte Pair Encoding (BPE) — Subword Tokenization

1
Start: split all words into individual characters.
2
Count all adjacent symbol pairs in the corpus.
3
Merge the most frequent pair into a new symbol; add to vocabulary.
4
Repeat until desired vocabulary size reached (or no more frequent pairs).

Handles rare and OOV (out-of-vocabulary) words gracefully. Used in GPT, BERT.

Large Language Models & Word Embeddings

Distributional Hypothesis

Foundational idea behind word embeddings:

"A word is characterized by the company it keeps." — Firth (1957)
Words appearing in similar contexts tend to have similar meanings (Harris 1954).
This underpins Word2Vec, GloVe, and all distributional semantic models.

Semantic Relationships in Embeddings

RelationshipDescriptionExample
SynonymySame meaningbig ≈ large
AntonymyOpposite meaninghot ↔ cold
SimilarityRelated domaindog, cat (both pets)
AnalogyStructural relationshipking – man + woman ≈ queen

Word Vector Analogy Arithmetic

vec("king") – vec("man") + vec("woman") ≈ vec("queen") vec("Paris") – vec("France") + vec("Italy") ≈ vec("Rome") This works because embeddings encode relational structure: Royalty vector ≈ king – man ≈ queen – woman Capital-of vector ≈ Paris – France ≈ Rome – Italy
One-hot limitation: One-hot vectors have cosine similarity = 0 for any two distinct words (orthogonal vectors). They carry zero semantic information — embeddings fix this by placing semantically similar words nearby in vector space.

Word Embeddings

Dense, low-dimensional vector representations that capture semantic similarity. Words used in similar contexts have similar vectors.

Word2Vec — Two Architectures

Skip-gram

  • Input: center word
  • Output: predict context words in window
  • Better for rare words
  • Slower to train

CBOW (Continuous Bag of Words)

  • Input: context words (averaged)
  • Output: predict center word
  • Better for frequent words
  • Faster to train

Skip-gram: Objective Function

Maximize: Σ Σ log P(o | c) t -m≤j≤m, j≠0 Where: P(o | c) = exp(u_o^T · v_c) / Σ_w exp(u_w^T · v_c) v_c = embedding of center word c (from input matrix V) u_o = embedding of context word o (from output matrix U) Denominator = softmax over all vocabulary words

Skip-gram with Negative Sampling (SGNS)

Solves the expensive softmax over full vocabulary by framing as binary classification:

y = 1 if (center, context) are real co-occurrences y = 0 for k randomly sampled "negative" (noise) pairs Loss = -[ log σ(u_o^T v_c) + Σ_k log σ(-u_k^T v_c) ] σ(x) = 1 / (1 + e^(-x)) (sigmoid function)
Key insight: SGNS turns an expensive multi-class problem (predict which of 50,000 words) into many cheap binary problems (is this pair real or noise?).

CBOW — Loss Function & Detail

CBOW input: average of context word vectors x = (1/2m) Σ v_w for w in context window of size m Predicted center word probability: P(c | context) = exp(u_c^T · x) / Σ_w exp(u_w^T · x) Loss: minimize NLL = -log P(c | context) Key: context vectors V_o are used as the word embeddings in CBOW (unlike Skip-gram where input vectors v_c are preferred)

Skip-gram Assumptions

  • Window independence: each context word is predicted independently given the center word.
  • Position independence: word order within the window does not matter — same prediction regardless of position.

GloVe — Global Vectors (Stanford)

An alternative to Word2Vec that uses global co-occurrence statistics from the full corpus matrix.

Word2VecGloVe
Training signalLocal context windowGlobal co-occurrence matrix X
MethodPredictive (neural, SGD)Matrix factorization (least squares)
Key advantageEfficient, streams dataCaptures global corpus statistics
Analogy taskStrongStrong (often similar performance)
GloVe objective: J = Σ_{i,j} f(X_{ij}) (w_i^T w_j + b_i + b_j − log X_{ij})^2 X_{ij} = number of times word j appears in context of word i f(x) = weighting function (caps contribution of very frequent pairs)

CBOW Neural Architecture

Layer 1: Embedding lookup → one vector per context word (dim = d) Layer 2: Lambda (mean pooling) → average of all context vectors x = mean(v_{w-m}, ..., v_{w-1}, v_{w+1}, ..., v_{w+m}) Layer 3: Dense (softmax) → probability distribution over vocabulary Output: P(center_word | context) = softmax(W · x + b)
  • Input: 2m one-hot context vectors (m = window size on each side)
  • Embedding matrix is shared — all context words use the same weight matrix
  • Pooling by averaging loses word order (hence "bag" of words)

Link Analysis — PageRank

Web as a Graph

  • Nodes: web pages
  • Edges: hyperlinks between pages (directed)
  • Links as Votes: a hyperlink from page A to page B is treated as a vote by A for B. Pages with more in-links are more important.
  • Weighted votes: a link from an important page is worth more than a link from an unimportant page → recursive definition → solved by PageRank.

Core Idea

A page is important if important pages link to it. PageRank distributes "rank" along edges, iterating until convergence.

Random Walk Interpretation

Imagine a random surfer starting at any page and following links uniformly at random:

  • At each step: with probability β, follow a random outgoing link.
  • With probability (1−β), teleport to any page chosen uniformly at random.
  • PageRank of a page = fraction of time the surfer spends on that page in the long run.
  • This fraction is the stationary distribution of the Markov chain defined by the Google Matrix A.
Stationary distribution π: satisfies π = A · π. Power iteration finds it by repeatedly multiplying until convergence: the rank vector r converges to π regardless of the starting distribution (as long as the graph is ergodic — ensured by the teleport term).

Stochastic Adjacency Matrix M

M[j][i] = 1 / out_degree(i) if there is an edge i → j M[j][i] = 0 otherwise Columns sum to 1 (column-stochastic). PageRank vector r satisfies: r = M · r (eigenvector of M with eigenvalue 1)

Power Iteration

1
Initialize: r = [1/N, 1/N, …, 1/N]
2
Multiply: r_new = M · r
3
Repeat until ||r_new − r||₁ < ε (convergence).

Converges in ~50 iterations for the web graph.

Problems and Fixes

ProblemCauseFix
Dead Ends Node with no outgoing edges — absorbs rank, never redistributes Teleport: treat dead-end as linking to all pages uniformly
Spider Traps Group of pages linking only to each other — traps all rank Random teleport with probability (1−β)

Google Matrix (The Full Formula)

A = β · M + (1 − β) · [1/N]_{N×N} r = A · r β ∈ [0.8, 0.9] — damping factor (probability of following a link) (1 − β) — probability of teleporting to a random page Interpretation: with prob β follow a link; with prob (1−β) jump anywhere.

Efficient Sparse Implementation (Tutorial 7 formula)

For sparse graphs, avoid computing the full (1−β)[1/N]J matrix. Instead use:

r^(t+1) = β · M · r^t + (1−β)/N This is algebraically equivalent to the Google Matrix form but avoids the dense N×N matrix — only the sparse M needs to be materialized. Steps in code: 1. links = [(src, dst), ...] as RDD 2. adj_list = links.groupByKey() 3. ranks = adj_list.map(lambda x: (x[0], 1.0/N)) 4. For each iteration: contribs = adj_list.join(ranks).flatMap( lambda (url, (links, rank)): [(link, rank/len(links)) for link in links] ) ranks = contribs.reduceByKey(add).mapValues( lambda rank: beta * rank + (1 - beta) / N )
Common exam question: Given a small graph, compute PageRank by hand using one or two iterations of power iteration. Know how to build matrix M and apply the Google formula.

Worked Example (3-node graph)

Nodes: A, B, C Edges: A→B, A→C, B→C, C→A A B C M = [ 0 0 1 ] ← who links TO A [ 1/2 0 0 ] ← who links TO B [ 1/2 1 0 ] ← who links TO C Start: r = [1/3, 1/3, 1/3]^T After 1 iter: r = M · r = [1/3, 1/6, 1/2]^T ... iterate until convergence

Graph Analytics

Graph Types

Undirected Graph

  • Edges have no direction (symmetric)
  • A–B = B–A
  • Examples: friendship networks, co-authorship
  • Degree: single value d(v)

Directed Graph (Digraph)

  • Edges have direction: A→B ≠ B→A
  • Examples: Twitter follow, web hyperlinks, citations
  • In-degree: number of edges pointing to v
  • Out-degree: number of edges leaving v

Unweighted Graph

  • All edges have equal weight (1)
  • Distance = hop count

Weighted Graph

  • Edges carry numeric weights (distances, capacities, similarities)
  • Examples: road networks (km), airline routes (cost)
  • Use Dijkstra's for shortest weighted paths
For directed graphs: Σ in-degree(v) = Σ out-degree(v) = |E| (every edge contributes 1 to in-degree of target and 1 to out-degree of source)

Clustering Coefficient

Measures how tightly a node's neighbors are connected to each other.

C(v) = (2 · m_v) / (d_v · (d_v − 1)) m_v = number of edges between v's neighbors d_v = degree of v d_v(d_v-1)/2 = maximum possible edges among neighbors Range: [0, 1]. 1 = all neighbors connected; 0 = no neighbor connections.

Centrality Measures

MeasureIntuitionFormulaLimitation
Closeness How close is this node to all others? C_clos(v) = (N−1) / Σ_x d(v,x) Undefined for disconnected graphs
Harmonic Closeness but handles disconnected components C_harm(v) = Σ_{x≠v} 1/d(v,x) —
Betweenness How often is this node on a shortest path? BC(v) = Σ_{s≠t} σ_st(v) / σ_st Expensive to compute (O(VE))

Betweenness Notation:

σ_st = total number of shortest paths from s to t σ_st(v) = number of those paths that pass through v BC(v) = Σ_{s≠v≠t} σ_st(v) / σ_st

Eigenvector Centrality — Power Iteration

A node is important if its neighbours are important. Solved iteratively:

c^(0) = [1, 1, ..., 1]^T (initialise uniformly) c^(t+1) = A · c^(t) / || A · c^(t) || (multiply by adjacency matrix, then normalise) Repeat until convergence. The converged vector is the eigenvector corresponding to the largest eigenvalue of A.
  • A[i][j] = 1 if edge exists between i and j (adjacency matrix)
  • Captures global influence — a low-degree node can have high eigenvector centrality if connected to hubs
  • PageRank is a variant of eigenvector centrality with teleportation

Edge Betweenness

Like node betweenness but for edges: how often does edge (u,v) appear on a shortest path?

EB(u,v) = Σ_{s≠t} σ_st(u,v) / σ_st Algorithm (BFS fractional counting): 1. Run BFS from each source s → find all shortest paths 2. For each target t: credit = 1 / σ_st 3. Distribute credit backwards along BFS tree edges 4. Edges on more shortest paths accumulate higher betweenness Use: Girvan-Newman community detection removes highest-EB edges iteratively.

Real-World Case Study: MSN Messenger Network (2006)

PropertyValue

Six Degrees of Separation: This confirmed the "six degrees" hypothesis — most pairs of people on Earth are connected through a surprisingly small number of intermediaries. Small-world property holds even at massive scale.

Centrality Limitations — Krackhardt's Kite Graph

No single centrality measure is "best" — different measures highlight different structural roles:

CentralityIdentifiesWho scores highest
DegreeMost directly connectedHub node (many neighbours)
ClosenessFastest to spread infoCentral node (short paths to all)
BetweennessControls information flowBridge / broker node
EigenvectorMost influential overallNode connected to other influential nodes
In the Krackhardt kite graph, these four measures give four different rankings. The "most important" node depends on what you mean by important.

Network Properties (Small-World & Scale-Free)

Small-World Network

  • Short average path lengths (6 degrees of separation)
  • High clustering coefficient
  • Example: social networks, WWW

Scale-Free Network

  • Degree distribution follows power law: P(k) ~ k^(-γ)
  • Most nodes have few connections; a few hubs have many
  • Example: internet topology, citation networks

Graph Connectivity

  • Connected graph: there exists a path between every pair of nodes.
  • Giant component: the largest connected subgraph — in web/social graphs this contains most nodes.
  • BFS (Breadth-First Search) is used to find connected components and compute shortest paths.
BFS from source s: 1. Mark s as visited, add to queue 2. While queue not empty: u = dequeue for each neighbour v of u: if v not visited: mark visited, enqueue v, distance[v] = distance[u]+1 Result: distance[] gives shortest path from s to all reachable nodes

Betweenness Centrality — Worked Example

Graph: A–B–C–D–E (linear chain) + B–D shortcut

Nodes: A, B, C, D, E Edges: A-B, B-C, C-D, D-E, B-D To compute BC(B): Count all (s,t) pairs where s≠B≠t For each pair, find σ_st (total shortest paths) and σ_st(B) (paths through B) Pair (A,C): paths A→B→C. σ=1, σ(B)=1 → contribution 1/1 = 1 Pair (A,D): paths A→B→D, A→B→C→D. σ=2 (B→D shorter, so only 1 shortest) Shortest: A→B→D. σ=1, σ(B)=1 → contribution 1 Pair (A,E): A→B→D→E. σ(B)=1 → contribution 1 Pair (C,E): C→D→E. σ(B)=0 → contribution 0 ... (sum all pairs) Key insight: nodes that are "bridges" between clusters have high betweenness. Removing high-betweenness nodes fragments the network.

Degree Distribution

  • Degree distribution P(k): probability that a randomly chosen node has degree k.
  • Random graphs → bell curve (Poisson-like).
  • Scale-free networks → power law: P(k) ~ k−γ (heavy tail — few hubs with very high degree).
Exam tip: Be able to compute clustering coefficient and betweenness centrality for a small hand-drawn graph.

Big Data Storage & NoSQL

ACID Properties (Traditional RDBMS)

PropertyMeaning
AtomicityAll operations in a transaction succeed, or none do (all-or-nothing)
ConsistencyDB moves from one valid state to another; all rules enforced
IsolationConcurrent transactions execute as if sequential; no interference
DurabilityCommitted transactions survive system failure

CAP Theorem

In a distributed system you can guarantee at most 2 of the 3:

C — Consistency

Every read returns the most recent write (or an error).

A — Availability

Every request gets a (non-error) response — but it may not be the most recent data.

P — Partition Tolerance

System continues operating despite network partitions (some messages between nodes dropped/delayed).
CombinationTrade-offExamples
CP Strong consistency; unavailable during network partition Traditional RDBMS (MySQL, PostgreSQL), HBase, Zookeeper
AP Always available; data may be stale Cassandra, DynamoDB, CouchDB (NoSQL)
CA Consistent and available; no partition tolerance (impossible in real distributed systems) Single-node RDBMS (no distribution)
Key rule: In practice, network partitions WILL happen. So real distributed systems must choose between C and A. CA is only theoretically possible on a single machine.

Strong vs Eventual Consistency

Strong Consistency

  • Read after write always returns the new value
  • All nodes see the same data at the same time
  • Higher latency; may block writes

Eventual Consistency

  • Given enough time, all nodes converge to same value
  • Reads may return stale data temporarily
  • Lower latency; always writable

BASE Model (NoSQL alternative to ACID)

LetterStands forMeaning
BBasically AvailableSystem always acknowledges requests (may return stale data)
SSoft StateState of system may change over time even without new input (due to propagation)
EEventually ConsistentSystem will reach consistency once propagation completes

Transactions & OLTP

Transaction: a sequence of operations executed as a single logical unit (all succeed or all fail).

BEGIN TRANSACTION UPDATE accounts SET balance = balance - 100 WHERE id = 'A'; UPDATE accounts SET balance = balance + 100 WHERE id = 'B'; COMMIT ← makes changes permanent -- or ABORT/ROLLBACK ← undoes all changes if something fails

OLTP (Online Transaction Processing)

  • Many short read/write transactions (ms latency)
  • High concurrency, frequent inserts/updates
  • Examples: banking, e-commerce, booking systems
  • Needs full ACID guarantees

OLAP (Online Analytical Processing)

  • Few complex read-heavy queries over large data
  • Aggregations, historical analysis
  • Examples: data warehouses, reporting dashboards
  • Spark, Hive, SparkSQL are OLAP tools

Challenges in Distributed RDBMS

Partitioning (Sharding)

  • Split data across multiple nodes by key range or hash
  • Improves write performance
  • Hurts reads that join across partitions

Replication

  • Copy same data to multiple nodes
  • Improves fault tolerance and read performance
  • Hurts write performance (must update all replicas)

Recommendation Systems

Utility Matrix

A utility matrix (or ratings matrix) has users as rows, items as columns, and entries are ratings. Most entries are empty — this is the sparsity problem.

Rating typeDescriptionExamples
ExplicitUser deliberately provides a ratingStars (1–5), thumbs up/down, written review
ImplicitInferred from behaviour without direct ratingWatch time, clicks, purchases, search history
Trade-off: Explicit ratings are accurate but sparse (users rarely rate). Implicit ratings are dense but noisy (watching something ≠ liking it).

Two Main Approaches

Content-Based Filtering

  • Recommend items similar to what the user liked before
  • Uses item features (genre, director, tags…)
  • No need for data from other users
  • Problem: Limited novelty — only recommends what's already known
  • Cold-start OK for items; cold-start bad for new users

Collaborative Filtering (CF)

  • Recommend based on what similar users liked
  • No item features needed — only ratings matrix
  • Problem: Cold-start for new users AND new items
  • Sparsity problem: most users rate very few items

Collaborative Filtering — Two Variants

User-Based CFItem-Based CF
IdeaFind users similar to target user; recommend what they likedFind items similar to those the user rated; recommend them
SimilarityBetween users (row vectors)Between items (column vectors)
StabilityUser preferences change → less stableItem characteristics stable → more stable
ScaleCostly as users growPre-compute item-item matrix (used by Amazon)

Similarity Metrics

Jaccard Similarity (for binary/implicit data): sim(A, B) = |A ∩ B| / |A ∪ B| Items rated by BOTH / Items rated by EITHER Range [0,1]; ignores rating values — good for implicit data Cosine Similarity: sim(u, v) = (u · v) / (||u|| · ||v||) Range: [-1, 1]. 1 = identical direction, 0 = orthogonal, -1 = opposite. Pearson Correlation (mean-centered cosine): sim(u, v) = Σ(r_ui - r̄_u)(r_vi - r̄_v) / (σ_u · σ_v) Adjusts for different user rating scales (harsh vs generous raters).

Matrix Factorization (Latent Factor Model)

Decompose ratings matrix R (users × items) into two lower-rank matrices:

R ≈ U · V^T U = (users × k) latent user matrix V = (items × k) latent item matrix k = number of latent factors (e.g., 50–200) Predicted rating: r̂_ui = u_i^T · v_j Optimize: minimize Σ (r_ui - u_i^T v_j)² + λ(||U||² + ||V||²) using SGD or ALS (Alternating Least Squares)

Baseline Estimate

Before applying CF, subtract user and item biases to get a better baseline prediction:

b_ui = μ + b_u + b_i μ = global average rating b_u = user bias (how harsh/generous the user rates vs average) b_i = item bias (how popular/unpopular the item is vs average) Improved CF prediction: r̂_ui = b_ui + (CF adjustment from neighbours)

MAEΣ|r̂ - r|/nMean absolute error
Precision@KRelevant in top-K / KWhat fraction of top-K are actually good?
Recall@KRelevant in top-K / total relevantDid we surface all good items?

ALS (Alternating Least Squares) — How it works

ALS is the standard algorithm for matrix factorization in Spark ML:

1
Initialize U and V (user/item matrices) with random values.
2
Fix V, solve for U by minimizing the loss (least squares on each user row).
3
Fix U, solve for V by minimizing the loss (least squares on each item column).
4
Repeat steps 2–3 until convergence (loss stops decreasing).
# Spark ML ALS implementation (Tutorial 9)
from pyspark.ml.recommendation import ALS
from pyspark.ml.evaluation import RegressionEvaluator

# Split data

# Build model

          userCol="userId", itemCol="movieId", ratingCol="rating")

# Evaluate with RMSE

evaluator = RegressionEvaluator(metricName="rmse",
                                 labelCol="rating",
                                 predictionCol="prediction")

Challenges

  • Cold Start: New user has no ratings; new item has no ratings. Fix: ask for initial preferences, use item content.
  • Sparsity: Typical ratings matrix is >99% empty. Fix: matrix factorization.
  • Scalability: Millions of users × millions of items. Fix: approximate nearest neighbors, distributed ALS.
  • Filter Bubble: Always recommending similar items → lack of diversity. Fix: inject serendipitous recommendations.

PageRank Algorithm Steps

1
Build column-stochastic matrix M from the graph adjacency.
2
Handle dead ends by teleporting (treat dead-end as linking to all pages).
3
Apply Google Matrix: A = βM + (1−β)(1/N)J.
4
Initialize r uniformly: r = [1/N, …, 1/N].
5
Iterate r = A·r until convergence.
