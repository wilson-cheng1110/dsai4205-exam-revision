Hashing Strategy for Sharding

Assign record with key k to node: node = h(k) % n h(k) = hash function (e.g., MD5, SHA) n = number of nodes Ensures even distribution of records across nodes. Problem: adding/removing nodes requires rehashing → Consistent Hashing solves this.

MapReduce

A programming model for processing large datasets in parallel across a cluster.

1
Input: A bag of (key, value) pairs.
2
Map: Each worker processes its chunk and emits intermediate (key, value) pairs.
3
Shuffle: All intermediate values with the same key are grouped together and sent to the same reducer.
4
Reduce: Aggregates the values for each key into a final result.
5
Output: Final (key, value) pairs written to storage.

Word Count Example (classic MapReduce)

Map: (doc_id, "hello world hello") → [("hello",1), ("world",1), ("hello",1)] Shuffle: ("hello", [1,1]), ("world", [1]) Reduce: ("hello", 2), ("world", 1)

HDFS (Hadoop Distributed File System)

  • Write-once, read-many model — files are immutable once written.
  • Fixed-size block partitioning — large files split into blocks (default 128 MB).
  • 3-way replication — each block replicated on 3 separate nodes for fault tolerance.
  • NameNode: master — stores metadata (file names, block locations).
  • DataNode: worker — stores actual data blocks.

Dask — Block Algorithms

Dask solves out-of-memory problems by splitting arrays/DataFrames into chunks and applying block algorithms:

Example: sum of a 4×4 array split into four 2×2 chunks Step 1: Each worker computes sum of its chunk independently chunk_A=10, chunk_B=26, chunk_C=18, chunk_D=42 Step 2: Combine chunk results → total = 10+26+18+42 = 96 Key principle: "Divide data → process chunk → combine results" Worker = distinct server / local machine / CPU core

Dask Task Graph

Every Dask operation builds a task graph before executing anything:

  • Nodes: individual tasks (e.g., compute chunk sum)
  • Edges: dependencies between tasks (output of one feeds input of next)
  • Graph is built lazily — no computation until .compute()
  • Visualise with result.visualize() (uses Graphviz)

Dask Chunk Specification (4 ways)

import dask.array as da

# 1. Same size for all dimensions
arr = da.ones((10000, 12000), chunks=1000)         # 1000 elements each dim

# 2. Different size per dimension
arr = da.ones((10000, 12000), chunks=(5000, 4000))

# 3. Dict specification per dimension
arr = da.ones((10000, 12000), chunks={0: 1000, 1: 2000})

# 4. Fully explicit block sizes (tuple of tuples)
arr = da.ones((10000, 12000), chunks=((5000,5000), (4000,4000,4000,4000)))

# Rechunk after creation (EXPENSIVE — avoid if possible)
arr.rechunk({0: -1, 1: 100})  # -1 = keep entire dimension as one chunk
# WARNING: 86ms → 7min 13s performance difference from bad rechunk choice
Chunk size guideline: 100 MB – 1 GB per chunk. Too small = task graph overhead. Too large = out-of-memory.

MapReduce — Detailed Walkthrough

Input: 3 documents spread across 3 mappers. Each runs independently in parallel.

Input documents: Mapper 1: "I often repeat repeat" Mapper 2: "I do not not" Mapper 3: "repeat after me" Map phase (each mapper emits (word, 1) for every word): Mapper 1: (I,1) (often,1) (repeat,1) (repeat,1) Mapper 2: (I,1) (do,1) (not,1) (not,1) Mapper 3: (repeat,1) (after,1) (me,1) Shuffle phase (group by key — sent across network): (I, [1, 1]) (often, [1]) (repeat,[1, 1, 1]) (do, [1]) (not, [1, 1]) (after, [1]) (me, [1]) Reduce phase (sum the list): (I, 2) (repeat, 3) (not, 2) ... Performance cost: ALL intermediate data written to HDFS between map and reduce (necessary for fault tolerance — but disk I/O makes it slow) Spark fix: keep intermediate data in MEMORY → 10–100× faster for iterative jobs

Core Concepts

RDD (Resilient Distributed Dataset)

  • Fundamental data structure in Spark
  • Immutable — create new RDD instead of modifying
  • Distributed — partitioned across cluster nodes
  • Fault-tolerant via data lineage (re-compute lost partitions)
  • Lazy evaluation — transformations not executed until an action is called

DataFrame

  • Organized into named columns (like a table)
  • Supports nested types: StructType, ArrayType, MapType
  • Optimized by Catalyst optimizer
  • DSL similar to Pandas
  • Also lazy — needs an action to execute

RDD — Data Lineage & Fault Tolerance

Problem: a node fails mid-computation — how to recover? RDD solution — Data Lineage: • Record the OPERATIONS (not the data) applied to each partition • Each RDD knows: "I was derived from parent X by operation F" • Recovery: re-run F on the surviving partitions of X → reconstruct lost partition • Cost: cheap — just replay operations; no expensive data replication needed Immutability principle: • RDDs CANNOT be modified — every operation creates a new RDD • This makes lineage tracking trivial (deterministic, no side effects)

Transformations vs Actions

CategoryDescriptionExamples
Transformation Lazy — returns a new RDD/DF; no computation yet map(), filter(), flatMap(), join(), union(), reduceByKey(), groupByKey(), distinct(), sortBy()
Action Eager — triggers computation, returns a value or writes data collect(), count(), show(), save(), first(), take(n)

Narrow vs Wide Transformations

Narrow

  • Each input partition → exactly one output partition
  • No network shuffle needed
  • Examples: filter, map, flatMap
  • Faster, can pipeline stages

Wide (Shuffle)

  • Multiple input partitions → output partitions
  • Requires network communication (shuffle)
  • Examples: groupByKey, sortBy, join
  • Expensive — stage boundary in DAG

map() vs mapValues() vs flatMapValues()

rdd = sc.parallelize([('a', [1, 2, 3]), ('b', [4, 5])])

# map() — applied to entire element (key + value together)
rdd.map(lambda x: (x[0], len(x[1]))).collect()
# [('a', 3), ('b', 2)]

# mapValues() — applied to VALUE only; key preserved unchanged
rdd.mapValues(lambda v: sum(v)).collect()
# [('a', 6), ('b', 9)]

# flatMapValues() — like mapValues but FLATTENS the list output
rdd.flatMapValues(lambda v: v).collect()
# [('a',1), ('a',2), ('a',3), ('b',4), ('b',5)]  ← each item becomes its own row

Partitioning & Join Optimisation

  • partitionBy(key) — partition RDD so all records with the same key land on the same node.
  • If two RDDs are partitioned by the same key, join() becomes a narrow transformation (no shuffle).
  • Mismatched partitioning → all-to-all network communication (expensive wide transformation).
  • Use HashPartitioner for key-based partitioning; RangePartitioner for sorted ranges.

Inverted Index (Take-Home Exercise)

Classic IR structure: word → list of document IDs that contain it.

# Input format: "D1 || word1 word2 word3 ..."
lines = spark.textFile("documents.txt")

inv_index = lines \
    .flatMap(lambda line: [
        (word, line.split("||")[0].strip())
        for word in line.split("||")[1].strip().split()
    ]) \
    .distinct() \
    .groupByKey() \
    .mapValues(lambda docs: sorted(list(docs))) \
    .sortByKey()

inv_index.collect()
# [('data', ['D1','D2']), ('science', ['D1','D3']), ...]

Text Feature Representations

One-Hot Encoding

  • Each word → binary vector of vocabulary size
  • Only the position of that word is 1, rest 0
  • Sparse; no notion of word similarity
  • Example: [Fish, Dog, Cat] → [1,0,0], [0,1,0], [0,0,1]

Bag of Words (BoW)

  • Count occurrences of each word in document
  • Word order is lost
  • Sparse representation
  • Can be weighted by TF-IDF for importance

TF-IDF — Full Formula with Worked Example

TF: how many times term t appears in document d. IDF: penalises terms common across many documents.

IDF(t, D) = log( (|D|+1) / (DF(t,D)+1) ) ← +1 = Laplace smoothing TF-IDF(t, d, D) = TF(t, d) × IDF(t, D) |D| = total docs; DF(t,D) = docs containing t; High TF-IDF = frequent in ONE doc, rare elsewhere

Worked Example (|D|=2)

D0="Python python Spark Spark", D1="Python SQL" TF(python,D0)=2 TF(spark,D0)=2 TF(python,D1)=1 TF(sql,D1)=1 DF(python)=2 DF(spark)=1 DF(sql)=1 IDF(python) = log(3/3) = 0 ← in BOTH docs → not discriminating IDF(spark) = log(3/2) ≈ 0.405 IDF(sql) = log(3/2) ≈ 0.405 TF-IDF(python,D0) = 2×0 = 0 TF-IDF(spark, D0) = 2×0.405 = 0.811 ← "spark" is keyword for D0 TF-IDF(sql, D1) = 1×0.405 = 0.405 ← "sql" is keyword for D1

Levenshtein (Edit) Distance

Minimum number of single-character edits to transform one string to another.

Operations: Insert, Delete, Substitute (each costs 1)

Recurrence: If s[i] == t[j]: dp[i][j] = dp[i-1][j-1] (no cost) Else: dp[i][j] = 1 + min( dp[i-1][j], # delete from s dp[i][j-1], # insert into s dp[i-1][j-1] # substitute )

Example: "kitten" → "sitting" = 3 edits (k→s, e→i, insert g)

Word Embeddings

Dense, low-dimensional vector representations that capture semantic similarity. Words used in similar contexts have similar vectors.

Word2Vec — Two Architectures

Skip-gram

  • Input: center word
  • Output: predict context words in window
  • Better for rare words
  • Slower to train

CBOW (Continuous Bag of Words)

  • Input: context words (averaged)
  • Output: predict center word
  • Better for frequent words
  • Faster to train

Skip-gram: Objective Function

Maximize: Σ Σ log P(o | c) t -m≤j≤m, j≠0 Where: P(o | c) = exp(u_o^T · v_c) / Σ_w exp(u_w^T · v_c) v_c = embedding of center word c (from input matrix V) u_o = embedding of context word o (from output matrix U) Denominator = softmax over all vocabulary words

Skip-gram with Negative Sampling (SGNS)

Solves the expensive softmax over full vocabulary by framing as binary classification:

y = 1 if (center, context) are real co-occurrences y = 0 for k randomly sampled "negative" (noise) pairs Loss = -[ log σ(u_o^T v_c) + Σ_k log σ(-u_k^T v_c) ] σ(x) = 1 / (1 + e^(-x)) (sigmoid function)
Key insight: SGNS turns an expensive multi-class problem (predict which of 50,000 words) into many cheap binary problems (is this pair real or noise?).

CBOW — Loss Function & Detail

CBOW input: average of context word vectors x = (1/2m) Σ v_w for w in context window of size m Predicted center word probability: P(c | context) = exp(u_c^T · x) / Σ_w exp(u_w^T · x) Loss: minimize NLL = -log P(c | context) Key: context vectors V_o are used as the word embeddings in CBOW (unlike Skip-gram where input vectors v_c are preferred)

Skip-gram — Three Formal Assumptions

  1. i.i.d. windows: each context window is an independent, identically distributed sample. Objective sums log-probabilities across all windows.
  2. Conditional independence: context words are predicted independently given the center word. P(o1,o2,…|c) = ∏ P(oi|c). No word depends on another context word.
  3. Position independence: word order / distance within the window is ignored — same prediction whether a word is 1 or m steps away from center.
These assumptions are clearly wrong linguistically — but they make optimization tractable and the resulting embeddings are still very good in practice.

CBOW Neural Architecture

Layer 1: Embedding lookup → one vector per context word (dim = d) Layer 2: Lambda (mean pooling) → average of all context vectors x = mean(v_{w-m}, ..., v_{w-1}, v_{w+1}, ..., v_{w+m}) Layer 3: Dense (softmax) → probability distribution over vocabulary Output: P(center_word | context) = softmax(W · x + b)
  • Input: 2m one-hot context vectors (m = window size on each side)
  • Embedding matrix is shared — all context words use the same weight matrix
  • Pooling by averaging loses word order (hence "bag" of words)

  • Weighted votes: a link from an important page is worth more than a link from an unimportant page → recursive definition → solved by PageRank.

Core Idea

A page is important if important pages link to it. PageRank distributes "rank" along edges, iterating until convergence.

Random Walk Interpretation

Imagine a random surfer starting at any page and following links uniformly at random:

  • At each step: with probability β, follow a random outgoing link.
  • With probability (1−β), teleport to any page chosen uniformly at random.
  • PageRank of a page = fraction of time the surfer spends on that page in the long run.
  • This fraction is the stationary distribution of the Markov chain defined by the Google Matrix A.
Stationary distribution π: satisfies π = A · π. Power iteration finds it by repeatedly multiplying until convergence: the rank vector r converges to π regardless of the starting distribution (as long as the graph is ergodic — ensured by the teleport term).

Stochastic Adjacency Matrix M

M[j][i] = 1 / out_degree(i) if there is an edge i → j M[j][i] = 0 otherwise Columns sum to 1 (column-stochastic). PageRank vector r satisfies: r = M · r (eigenvector of M with eigenvalue 1)

Power Iteration

1
Initialize: r = [1/N, 1/N, …, 1/N]
2
Multiply: r_new = M · r
3
Repeat until ||r_new − r||₁ < ε (convergence).

Converges in ~50 iterations for the web graph.

Problems and Fixes

ProblemCauseFix
Dead Ends Node with no outgoing edges — absorbs rank, never redistributes Teleport: treat dead-end as linking to all pages uniformly
Spider Traps Group of pages linking only to each other — traps all rank Random teleport with probability (1−β)

Dead-End Rank Redistribution (detail)

Dead-end pages cause rank leakage — their accumulated rank is never passed on. Two complementary fixes:

  • Per-iteration re-insertion: after each r = M·r step, sum leaked rank from dead-end nodes (those contributing 0 to the next vector) and redistribute it equally to all N pages: leaked = (1 − sum(r_new)) / N
  • Google Matrix (teleport): the (1−β)/N term achieves the same effect globally for all nodes, not just dead ends.

PageRank at Scale

Web scale (1998 reference): ~1 billion pages in the web graph Rank vector stores one float64 per page → ~8 GB for one copy of r Two copies needed per iteration (r_old + r_new) → ~16 GB minimum Why distributed matters: Single machine cannot hold 16 GB rank vector in 1998 RAM Distributed (Spark/Hadoop): partition adjacency list + rank vector across nodes Communication cost: only rank contributions (src→dst pairs) cross nodes each iteration

Google Matrix (The Full Formula)

A = β · M + (1 − β) · [1/N]_{N×N} r = A · r β ∈ [0.8, 0.9] — damping factor (probability of following a link) (1 − β) — probability of teleporting to a random page Interpretation: with prob β follow a link; with prob (1−β) jump anywhere.

Efficient Sparse Implementation (Tutorial 7 formula)

For sparse graphs, avoid computing the full (1−β)[1/N]J matrix. Instead use:

r^(t+1) = β · M · r^t + (1−β)/N This is algebraically equivalent to the Google Matrix form but avoids the dense N×N matrix — only the sparse M needs to be materialized. Steps in code: 1. links = [(src, dst), ...] as RDD 2. adj_list = links.groupByKey() 3. ranks = adj_list.map(lambda x: (x[0], 1.0/N)) 4. For each iteration: contribs = adj_list.join(ranks).flatMap( lambda (url, (links, rank)): [(link, rank/len(links)) for link in links] ) ranks = contribs.reduceByKey(add).mapValues( lambda rank: beta * rank + (1 - beta) / N )
Common exam question: Given a small graph, compute PageRank by hand using one or two iterations of power iteration. Know how to build matrix M and apply the Google formula.

Dead-End Full Algorithm - Leaked Rank Re-insertion

while not converged:
    for j in all_pages:
        if in_degree(j) > 0:
            r_new[j] = sum(beta * r[i] / out_degree(i) for i in predecessors(j))
        else:
            r_new[j] = 0

    leaked = 1 - sum(r_new)    # equals 1-beta when no dead-ends
    for j in all_pages:
        r_new[j] += leaked / N  # redistribute uniformly

Clustering Coefficient

Measures how tightly a node's neighbors are connected to each other.

Local (per-node): C(v) = (2 · m_v) / (d_v · (d_v − 1)) m_v = number of edges between v's neighbors d_v = degree of v (need d_v ≥ 2, else C(v) = 0) Range: [0, 1]. 1 = all neighbors connected; 0 = no neighbor connections. Eigenvector Centrality — Power Iteration

A node is important if its neighbours are important. Solved iteratively:

c^(0) = [1, 1, ..., 1]^T (initialise uniformly) c^(t+1) = A · c^(t) / || A · c^(t) || (multiply by adjacency matrix, then normalise) Repeat until convergence. The converged vector is the eigenvector corresponding to the largest eigenvalue of A.
  • A[i][j] = 1 if edge exists between i and j (adjacency matrix)
  • Captures global influence — a low-degree node can have high eigenvector centrality if connected to hubs
  • PageRank is a variant of eigenvector centrality with teleportation

Edge Betweenness

Like node betweenness but for edges: how often does edge (u,v) appear on a shortest path?

EB(u,v) = Σ_{s≠t} σ_st(u,v) / σ_st Algorithm (BFS fractional counting): 1. Run BFS from each source s → find all shortest paths 2. For each target t: credit = 1 / σ_st 3. Distribute credit backwards along BFS tree edges 4. Edges on more shortest paths accumulate higher betweenness Use: Girvan-Newman community detection removes highest-EB edges iteratively.

In-Degree Centrality = Majority Voting

In-degree centrality = (number of incoming edges) / (N-1). Equivalent to majority voting: each in-neighbor casts one vote. Fast and simple; ignores voter quality.

Closeness Centrality - Disconnected Graph Failure

Eigenvector Centrality - 4-Step Power Iteration Pseudocode

Step 1: c^(0) = [1, 1, ..., 1]^T, k = 1 Step 2: c^(k) = A . c^(k-1) (matrix-vector multiply) Step 3: c^(k) = c^(k) / ||c^(k)||_2 (L2 normalise) Step 4: if ||c^(k) - c^(k-1)||_1 < tol: STOP; else k += 1, go to Step 2 Intuition: "each node gives away its score to its successors; after many iterations the score stabilises to the dominant eigenvector." Which Centrality to Use?

Krackhardt kite graph: degree, closeness, and betweenness each pick a DIFFERENT "most important" node. No centrality measure is universally best.
  • Rankings only order nodes; they don't quantify the gap between importance levels.
  • Generalization failure: The measure that identifies the top node doesn't necessarily rank the rest correctly.
  • Study of 63 networks (Vignery and Laurier, 2020): no single measure dominates.

Edge Betweenness - Forward BFS + Backward Fractional Flow Algorithm

Girvan-Newman edge betweenness calculation (3 phases per source node s): Phase 1 - BFS from s, count shortest paths sigma(s -> v): sigma(s) = 1 sigma(s -> v) = sum of sigma(s -> parent) for each BFS parent of v Phase 2 - Backward pass (leaves to root), compute edge flow credit: Each TARGET node t gets flow = 1 Edge (u, v) where u is parent of v in BFS tree: credit(u->v) = sigma(s->u) / sigma(s->v) x (1 + sum of child edge credits of v) Phase 3 - Accumulate over all source nodes, divide by 2 (undirected counts each path twice) Example (linear graph A-B-C-D-E from L8 slides): Node A: lies on shortest paths between no two nodes -> BC(A) = 0 Node B: lies on paths from A to {C,D,E} -> BC(B) = 3 Node C: lies on paths from {A,B} to {D,E} -> BC(C) = 4 (= 2 x 2 pairs) Node D: BC(D) = 3, Node E: BC(E) = 0 For node B with shortcut edge B-D added: BC(B) = 3.5 (some paths split between B-C and B-D routes)

CAP Theorem

In a distributed system you can guarantee at most 2 of the 3:

C — Consistency

Every read returns the most recent write (or an error).

A — Availability

Every request gets a (non-error) response — but it may not be the most recent data.

P — Partition Tolerance

System continues operating despite network partitions (some messages between nodes dropped/delayed).
CombinationTrade-offExamples
CP Strong consistency; unavailable during network partition Traditional RDBMS (MySQL, PostgreSQL), HBase, Zookeeper
AP Always available; data may be stale Cassandra, DynamoDB, CouchDB (NoSQL)
CA Consistent and available; no partition tolerance (impossible in real distributed systems) Single-node RDBMS (no distribution)
Key rule: In practice, network partitions WILL happen. So real distributed systems must choose between C and A. CA is only theoretically possible on a single machine.

Strong vs Eventual Consistency

Strong Consistency

  • Read after write always returns the new value
  • All nodes see the same data at the same time
  • Higher latency; may block writes

Eventual Consistency

  • Given enough time, all nodes converge to same value
  • Reads may return stale data temporarily
  • Lower latency; always writable

Utility Matrix

A utility matrix (or ratings matrix) has users as rows, items as columns, and entries are ratings. Most entries are empty — this is the sparsity problem.

Given m users and n items: R is an m × n matrix where R[i,j] = rating of user i for item j Most entries are UNKNOWN — goal is to estimate them Typical sparsity: >99% of entries are missing (Netflix, Amazon)
Rating typeDescriptionExamplesProblem
ExplicitUser deliberately provides a ratingStars (1–5), thumbs up/down, written reviewSparse — most users don't bother rating
ImplicitInferred from behaviour without direct ratingWatch time, clicks, purchases, search historyCan only infer likes; cannot infer dislikes (not buying ≠ dislike)
Implicit feedback asymmetry: You can observe that a user bought/clicked something (positive signal). But if they did NOT buy something, you don't know if they dislike it, haven't seen it, or can't afford it. This makes implicit feedback fundamentally harder to learn from.

Two Main Approaches

Content-Based Filtering

  • Recommend items similar to what the user liked before
  • Uses item features (genre, director, tags…)
  • No need for data from other users
  • Problem: Limited novelty — only recommends what's already known
  • Cold-start OK for items; cold-start bad for new users

Collaborative Filtering (CF)

  • Recommend based on what similar users liked
  • No item features needed — only ratings matrix
  • Problem: Cold-start for new users AND new items
  • Sparsity problem: most users rate very few items
CF Key Assumption: "If user A has the same opinion as user B on item X, then A is more likely to have B's opinion on a different item Y." — Users with similar past behaviour will have similar future preferences.

Similarity Metrics

Jaccard Similarity (for binary/implicit data): sim(A, B) = |A ∩ B| / |A ∪ B| Items rated by BOTH / Items rated by EITHER Range [0,1]; ignores rating values — good for implicit data Cosine Similarity: sim(u, v) = (u · v) / (||u|| · ||v||) Range: [-1, 1]. 1 = identical direction, 0 = orthogonal, -1 = opposite. Pearson Correlation (mean-centered cosine): sim(u, v) = Σ(r_ui - r̄_u)(r_vi - r̄_v) / (σ_u · σ_v) Adjusts for different user rating scales (harsh vs generous raters).

KNN-Based Collaborative Filtering

The simplest CF approach: find the k most similar users (or items) and use their ratings to predict.

User-based prediction for user u on item i: r̂_ui = r̄_u + Σ_{v ∈ kNN(u)} sim(u,v) · (r_vi − r̄_v) ───────────────────────────────────────── Σ_{v ∈ kNN(u)} |sim(u,v)| r̄_u = mean rating of user u (normalises for harsh/generous raters) kNN(u) = k nearest neighbours of u by similarity metric sim(u,v) = cosine, Pearson, or Jaccard similarity Steps: 1. Compute similarity between target user u and ALL other users 2. Pick top-k most similar users (k typically 20–50) 3. Weighted average of their ratings for item i
KNN limitation: O(N) similarity computation per prediction — does not scale to millions of users. Matrix factorization (ALS) is far more scalable.

Matrix Factorization (Latent Factor Model)

Decompose ratings matrix R (users × items) into two lower-rank matrices:

R ≈ U · V^T U = (users × k) latent user matrix V = (items × k) latent item matrix k = number of latent factors (e.g., 50–200) Predicted rating: r̂_ui = u_i^T · v_j Optimize: minimize Σ (r_ui - u_i^T v_j)² + λ(||U||² + ||V||²) using SGD or ALS (Alternating Least Squares)

Baseline Estimate

Before applying CF, subtract user and item biases to get a better baseline prediction:

b_ui = μ + b_u + b_i μ = global average rating b_u = user bias (how harsh/generous the user rates vs average) b_i = item bias (how popular/unpopular the item is vs average) Improved CF prediction: r̂_ui = b_ui + (CF adjustment from neighbours)

ALS (Alternating Least Squares) — How it works

ALS is the standard algorithm for matrix factorization in Spark ML:

1
Initialize U and V (user/item matrices) with random values.
2
Fix V, solve for U by minimizing the loss (least squares on each user row).
3
Fix U, solve for V by minimizing the loss (least squares on each item column).
4
Repeat steps 2–3 until convergence (loss stops decreasing).

Challenges

  • Cold Start: New user has no ratings; new item has no ratings. Fix: ask for initial preferences, use item content.
  • Sparsity: Typical ratings matrix is >99% empty. Fix: matrix factorization.
  • Scalability: Millions of users × millions of items. Fix: approximate nearest neighbors, distributed ALS.
  • Filter Bubble: Always recommending similar items → lack of diversity. Fix: inject serendipitous recommendations.

The Long Tail — Origin

Physical stores are constrained by shelf space → they stock only the top-N most popular items. Online stores have no shelf-space limit, so they can profitably serve the "long tail" of niche items. The long tail items together often outsell the hits (Amazon: ~57% of sales from long-tail titles).

User-Based CF — Explicit Weighted Average Formula

Predict rating of user u for item i: r̂_u,i = ( Σ_{v} sim(u,v) · r_{v,i} ) / ( Σ_{v} |sim(u,v)| ) where the sums are over all neighbours v who have rated item i. This is the raw (non-baseline-adjusted) version. The baseline-adjusted version (shown elsewhere) adds b_ui.

Item-Item CF — Numerical Worked Example

Items 3 and 4 are similar to target item 5: sim(item3, item5) = 0.41, r_{user,3} = 2 sim(item4, item5) = 0.59, r_{user,4} = 3 Predicted rating: r̂ = (0.41 × 2 + 0.59 × 3) / (0.41 + 0.59) = (0.82 + 1.77) / 1.00 = 2.59 ≈ 2.6 Key insight: items change less than users → item-item similarity is more stable → item-item CF is preferred over user-user CF at production scale (Amazon uses this)

def nearest_words(word, w2i, i2w, k=5):
    idx = w2i[word]
    sims = sim_matrix[idx]              # similarity to all words
    top_k = np.argsort(sims)[::-1][1:k+1]  # skip self (index 0)
    return [(i2w[i], sims[i]) for i in top_k]

def compare_words(w1, w2, w2i):
    i1, i2 = w2i[w1], w2i[w2]
    return float(np.dot(W_norm[i1], W_norm[i2]))   # cosine similarity

Exam Review — PageRank Numerical Traces

Spider Trap Example (nodes y, a, m; β=0.85)

Graph: y→y, y→a, a→m, m→a (m is a spider trap — links only within {a,m}) Power iteration (β=0.85, start r=[1/3, 1/3, 1/3]): The spider trap {a,m} absorbs rank over iterations. Converged solution (approximate): r_y ≈ 7/33 ≈ 0.212 r_a ≈ 5/33 ≈ 0.152 r_m ≈ 21/33 ≈ 0.636 ← trap absorbs most rank Fix: teleportation (Google Matrix) redistributes 1-β=0.15 uniformly

Dead End Example

A dead end node has no out-links → its rank is never redistributed → rank leaks out. Fix: treat dead-end as linking to ALL pages with equal probability 1/N. Equivalently: add row (1/N, 1/N, ... 1/N) for dead-end columns in M. After applying teleportation (β=0.85): r_new = β·M·r + (1-β)/N This ensures rank is always conserved (sums to 1) every iteration.

Edit Distance — DP Table (BUT → ANT)

Levenshtein distance: BUT → ANT "" A N T "" 0 1 2 3 B 1 1 2 3 U 2 2 2 3 T 3 3 3 2 ← answer = 2 Operations: substitute B→A (cost 1), substitute U→N (cost 1), T→T (match, cost 0) Recurrence: dp[i][j] = 0 if s1[i]==s2[j], else 1 + min(dp[i-1][j], dp[i][j-1], dp[i-1][j-1])

BPE — Full Worked Example

Byte-Pair Encoding (subword tokenization): Start: character-level vocab with end-of-word marker Corpus: "low low low lowest newest" Initial: {l,o,w,</w>: 3}, {l,o,w,e,s,t,</w>: 1}, {n,e,w,e,s,t,</w>: 1} Step 1: Find most frequent pair → (l, o) appears 5× → merge to "lo" Step 2: Find most frequent pair → (lo, w) appears 5× → merge to "low" Step 3: Find most frequent pair → (low, </w>) appears 3× → merge to "low</w>" ... continue until vocab size reached Final vocab includes: {"l","o","w","e","s","t","n","lo","low","low</w>","est","..."} Tokenization: "lowest" → "low" + "est</w>" (uses longest matching merge rule)

Betweenness Centrality — Numerical Example

Betweenness centrality of node v: BC(v) = Σ_{s≠v≠t} σ_st(v) / σ_st where σ_st = number of shortest paths from s to t σ_st(v) = those paths that pass through v Example (5-node path graph 1-2-3-4-5): Node 3 lies on ALL paths between {1,2} and {4,5} BC(3) = (2×2) / (2×2) = 1.0 (normalised) For a star graph (center node C): C lies on every inter-leaf shortest path BC(C) = (N-1)(N-2)/2 / ((N-1)(N-2)/2) = 1.0 (normalized)

ROUGE Evaluation Metrics (Text Summarization)

ROUGE = Recall-Oriented Understudy for Gisting Evaluation Range [0, 1]; higher = better. Three variants: ROUGE-N: n-gram overlap (N=1 unigrams, N=2 bigrams) Recall = matching n-grams / n-grams in reference Precision = matching n-grams / n-grams in candidate F1 = 2 x P x R / (P + R) ROUGE-L: Longest Common Subsequence (word order matters but gaps allowed) Recall = LCS length / reference length Precision = LCS length / candidate length ROUGE-S: skip-bigram overlap (not required for this course)

ROUGE Worked Examples

Candidate: "Summarization is cool" Reference: "Summarization is beneficial and cool" ROUGE-1 (unigrams): matching = {Summarization, is, cool} = 3 Recall = 3/5 = 0.60 Precision = 3/3 = 1.00 F1 = 2x0.60x1.00 / (0.60+1.00) = 0.75 ROUGE-2 (bigrams): candidate bigrams = {Summ-is, is-cool} Reference bigrams = {Summ-is, is-beneficial, beneficial-and, and-cool} Matching = {Summ-is} = 1 Recall = 1/4 = 0.25, Precision = 1/2 = 0.50 F1 = 2x0.25x0.50 / (0.25+0.50) = 0.33 ROUGE-L: LCS = {Summarization, is, cool} length=3 Recall = 3/5 = 0.60, Precision = 3/3 = 1.00, F1 = 0.75 Rule: when multiple references exist, take the BEST score across references.

PageRank Algorithm Steps

1
Build column-stochastic matrix M from the graph adjacency.
2
Handle dead ends by teleporting (treat dead-end as linking to all pages).
3
Apply Google Matrix: A = βM + (1−β)(1/N)J.
4
Initialize r uniformly: r = [1/N, …, 1/N].
5
Iterate r = A·r until convergence.
